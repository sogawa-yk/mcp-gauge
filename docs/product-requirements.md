# プロダクト要求定義書 (Product Requirements Document)

## プロダクト概要

### 名称
**MCP Gauge** - コーディングエージェントによるMCPサーバー自律開発を実現する計測・評価MCPサーバー

### プロダクトコンセプト
- **エージェント自律開発の実現**: コーディングエージェント（Claude Code等）がMCPサーバーを開発する際に、自らMCP Gaugeを使ってテスト・評価・チューニングまで自律的に完結できる
- **LLMを含むE2Eフィードバックループ**: MCPサーバーは最終的にLLMが使うものであり、LLMが実際にどう使うかを計測してこそ意味のあるテストになる。このフィードバックを構造化データでエージェントに返す
- **MCPの内側から観測**: クライアント実装に依存せず、MCPプロトコルレベルでツール呼び出しをトレース・計測する

### プロダクトビジョン
MCPサーバーの開発は「実装 → 手動でエージェントに使わせてみる → 問題を発見 → 修正」という人間依存のサイクルに縛られている。MCP Gaugeは、コーディングエージェント自身がこのサイクルを自律的に回せるようにする。エージェントがMCPサーバーを実装し、MCP Gaugeでテストを実行し、結果を解釈してツール説明文やパラメータ設計を改善する——この「実装→テスト→チューニング」ループを完全に自動化する。人間はゴールの設定とレビューに集中できる。

### 目的
- コーディングエージェントがMCPサーバーの品質を自ら計測・評価できるツールを提供する
- 「実装→テスト→チューニング」の自律開発ループを実現する
- LLMの実利用パターンに基づく定量的フィードバックをエージェントが解釈可能な形式で返す
- MCPサーバーの変更に対する回帰テストをエージェント主導で自動化する
- 人間の介入なしにツール説明文・Prompts・パラメータ設計の最適化を可能にする

## ターゲットユーザー

### プライマリーユーザー: コーディングエージェント（Claude Code等）
MCP Gaugeの主たる利用者はコーディングエージェント自身である。エージェントはMCPサーバーの開発タスクを遂行する過程で、MCP Gaugeのツールを呼び出してテスト・評価・チューニングを行う。

- **利用コンテキスト**: MCPサーバーの新規開発・機能追加・修正時に、エージェントが自律的に品質を検証する
- **求められるインターフェース**: エージェントが解釈・判断・次のアクションに繋げられる構造化データ（JSON）での結果返却
- **自律開発ループ**:
  1. MCPサーバーのコードを実装・修正する
  2. MCP Gaugeでリンティング・E2Eテストを実行する
  3. 結果を解釈し、ツール説明文やパラメータ設計を改善する
  4. 再テストで改善を確認する
  5. 人間にレビューを依頼する

### セカンダリーユーザー: MCPサーバー開発者（人間）
エージェントが生成したテスト結果やレポートをレビューし、最終的な品質判断を行う。

- **ペルソナ**: 佐藤健一（32歳、MCPサーバー開発者）
- Claude Code等のコーディングエージェントにMCPサーバー開発を指示する
- **現在の課題**: エージェントにMCPサーバー開発を任せても、テスト・チューニングの段階で人間がループに入る必要がある。「実装してテストして」と指示しても、エージェントにはMCPサーバーの品質を評価する手段がない
- **期待する解決策**: 「MCP Gaugeを使ってテスト・チューニングまで完了して」と指示するだけで、エージェントが自律的に品質を担保できる

## 成功指標（KPI）

### プライマリーKPI
- 自律開発ループの完結率: 80%以上
  - 計測方法: MCP Gauge導入プロジェクトのE2Eテスト実行ログを集計
  - 計測対象: F3（シナリオベースE2Eテスト）のスイート実行履歴
  - 計測タイミング: 各プロジェクトの初期テスト完了後から起算して1ヶ月
- MCPサーバー開発プロジェクトへの導入数: 10プロジェクト（公開後6ヶ月）
  - 計測方法: GitHub上での依存・設定ファイルでの利用確認
- GitHub Stars: 500（公開後6ヶ月）

### セカンダリーKPI
- エージェントによるチューニング後、LLMのツール呼び出し成功率が10%以上改善
- エージェントによるPrompts改善後のステップ数削減: 平均20%削減
- 人間のレビュー工数削減: MCPサーバー開発における手動テスト時間を50%削減

## 機能要件

### コア機能（MVP）

#### F1: ツール説明文リンティング

**ユーザーストーリー**:
コーディングエージェントとして、自分が実装したMCPサーバーのツール説明文の品質を即座にチェックするために、LLM呼び出し不要な静的リンティング機能が欲しい

**受け入れ条件**:
- [ ] 対象MCPサーバーに接続し、全ツールのdescriptionとinputSchemaを取得できる
- [ ] 曖昧な表現（「適切な値を指定」「必要に応じて」等）を検出し警告する
- [ ] 必須パラメータに説明が不足している場合に警告する
- [ ] 任意パラメータのデフォルト値が説明に記載されていない場合に警告する
- [ ] 戻り値の構造がdescriptionに記載されていない場合に警告する
- [ ] 結果をJSON形式で返し、エージェントが問題箇所と改善アクションを特定できる
- [ ] 重大度レベル（error / warning / info）で分類される
- [ ] 各警告に対して具体的な改善提案（推奨される記述例）を含む

**優先度**: P0（必須）

**エージェント自律開発での利用フロー**:
1. エージェントがMCPサーバーのツールを実装する
2. `gauge_lint` でリンティングを実行する
3. 結果のerror/warningを読み取り、ツール説明文を修正する
4. 再度リンティングを実行し、警告が解消されたことを確認する

#### F2: LLM行動トレーシング

**ユーザーストーリー**:
コーディングエージェントとして、テスト対象MCPサーバーに対するLLMのツール呼び出しパターンを記録・分析するために、MCPプロトコルレベルのトレーシング機能が欲しい

**受け入れ条件**:
- [ ] テスト対象MCPサーバーへのツール呼び出しをプロキシし、リクエスト/レスポンスを記録する
- [ ] タスク完了までのツール呼び出し回数を計測する
- [ ] 同一ツールの冗長な再呼び出しを検出する
- [ ] ツール間の呼び出し順序を記録する
- [ ] エンドツーエンドの所要時間を計測する
- [ ] エラー発生時のリカバリに要した追加ステップ数を計測する
- [ ] タスクの成功/失敗を判定する
- [ ] トレースデータをJSON形式で返し、エージェントが問題パターンを特定できる

**優先度**: P0（必須）

**エージェント自律開発での利用フロー**:
1. エージェントがリンティング修正後のMCPサーバーを起動する
2. `gauge_trace_start` でトレースセッションを開始する
3. テスト用の操作を対象サーバーに対して実行する
4. `gauge_trace_stop` でセッションを終了し、サマリーを取得する
5. 冗長呼び出しやリカバリステップの数値を確認し、ツール説明文を改善する

#### F3: シナリオベースE2Eテスト

**ユーザーストーリー**:
コーディングエージェントとして、定義したシナリオに従ってLLMにMCPサーバーを操作させ、結果を検証するために、シナリオベースのE2Eテスト実行機能が欲しい

**受け入れ条件**:
- [ ] YAML/JSON形式でテストシナリオを定義できる
- [ ] シナリオにはタスク指示、期待されるツール呼び出しパターン、成功条件を含められる
- [ ] LLMにシナリオのタスク指示を与え、テスト対象MCPサーバーのツールを使って自動実行できる
- [ ] 実行結果をトレースデータとして記録する
- [ ] 成功条件（最大ステップ数、必須ツール呼び出し、禁止ツール呼び出し等）に基づいて合否判定する
- [ ] 複数シナリオを一括実行し、結果サマリーを返す
- [ ] 失敗シナリオについて、失敗原因の構造化データ（どのステップで何が期待と異なったか）を返す

**優先度**: P0（必須）

**エージェント自律開発での利用フロー**:
1. エージェントがMCPサーバーを実装する
2. テストシナリオを作成する（または既存シナリオを使う）
3. `gauge_run_scenario` / `gauge_run_suite` でE2Eテストを実行する
4. 失敗した場合、失敗原因を分析し、MCPサーバーのコードやツール説明文を修正する
5. 再テストで合格を確認する

#### F4: Prompts品質評価

**ユーザーストーリー**:
コーディングエージェントとして、PromptsやツールdescriptionがLLMのツール利用効率にどう影響するかを評価するために、変更前後の比較機能が欲しい

**受け入れ条件**:
- [ ] 同一シナリオをPrompts/description変更前後で実行し、ステップ数を比較できる
- [ ] LLMが誤ったパラメータでツールを呼んだ回数を計測する
- [ ] ツールディスカバリ後にLLMが使わなかったツールを検出する
- [ ] 比較結果を構造化データで返し、エージェントが「改善した/悪化した」を判定できる
- [ ] 統計的に有意な差異を判定するため、複数回実行の集計に対応する（LLMの応答非決定性により1回比較ではノイズが大きく判断不能なためMVP必須）

**優先度**: P0（必須）

**エージェント自律開発での利用フロー**:
1. エージェントが現在のdescriptionでベースラインテストを実行する
2. ツール説明文を改善する
3. `gauge_compare` で改善前後を比較する
4. 悪化した指標があれば説明文を再調整する
5. 全指標が改善または維持されたことを確認する

### 拡張機能（Post-MVP）

#### F5: セキュリティ・安全性検証

**ユーザーストーリー**:
コーディングエージェントとして、実装したMCPサーバーにセキュリティ上の問題がないかを自動検証するために、安全性チェック機能が欲しい

**受け入れ条件**:
- [ ] 破壊的操作（apply/destroy等）が前提条件なしに呼ばれる経路を探索する
- [ ] ユーザー入力がCLIコマンド等にそのまま渡されるインジェクションパターンを検出する
- [ ] ツール群の組み合わせで危険な操作に到達できるパスをファジングする
- [ ] 権限境界の検証（読み取り専用フローから書き込み操作への到達を検出）

**優先度**: P1（重要）

#### F6: シナリオレコーダー / リプレイ

**ユーザーストーリー**:
コーディングエージェントとして、手動テスト中のエージェント対話セッションを記録し、回帰テストとして再利用するために、セッション記録・リプレイ機能が欲しい

**受け入れ条件**:
- [ ] 実セッションのツール呼び出しシーケンスを記録できる
- [ ] 記録したシナリオをMCPサーバー変更後にリプレイして回帰テストできる
- [ ] 記録されたシナリオを編集してテストケースとして保存できる

**優先度**: P1（重要）

#### F7: コスト推定・最適化レコメンデーション

**ユーザーストーリー**:
コーディングエージェントとして、ツール設計の変更がトークン消費に与える影響を把握し、コスト効率の良い設計を選択するために、コスト可視化機能が欲しい

**受け入れ条件**:
- [ ] 各シナリオの推定トークン消費量を記録する
- [ ] Prompts/ツール設計の変更がコストに与える影響を可視化する
- [ ] 冗長な戻り値の検出と要約形式への変更をレコメンドする

**優先度**: P1（重要）

#### F8: コントラクトテスト

**ユーザーストーリー**:
コーディングエージェントとして、MCPサーバー間の依存関係における互換性を自動検証するために、コントラクトテスト機能が欲しい

**受け入れ条件**:
- [ ] サーバー間のインターフェース契約（入出力スキーマ）を定義・検証できる
- [ ] 上流サーバーの変更が下流サーバーに与える影響を検出する

**優先度**: P2（できれば）

#### F9: マルチクライアント互換性テスト

**ユーザーストーリー**:
コーディングエージェントとして、異なるLLMモデルでの互換性を検証するために、マルチクライアントテスト機能が欲しい

**受け入れ条件**:
- [ ] 同一シナリオをClaude / GPT / Gemini等で実行し結果を比較できる
- [ ] モデル固有のツール呼び出しの癖による失敗パターンを検出する
- [ ] 互換性レポートを生成する

**優先度**: P2（できれば）

### MCPインターフェース

MCP Gauge自体がMCPサーバーとして以下のツールを提供する。全ての結果はエージェントが解釈・判断可能な構造化JSON形式で返す:

```
# ツール説明文リンティング
gauge_lint(server_command, server_args?) → リンティング結果
  # エージェントは結果を見てdescriptionを修正し、再実行する

# トレーシング開始・停止
gauge_trace_start(server_command, server_args?) → trace_id
gauge_trace_stop(trace_id) → トレースサマリー（メトリクス付き）

# シナリオ実行
gauge_run_scenario(server_command, scenario, server_args?) → 実行結果（合否+詳細）
gauge_run_suite(server_command, suite_path, server_args?) → スイート結果

# Prompts/Description比較
gauge_compare(server_command, scenario, baseline_trace_ids, server_args?) → 比較結果（改善/悪化の判定付き。baseline_trace_idsは複数指定で統計集計に対応）

# レポート
gauge_report(trace_ids) → 統合レポート
```

## 非機能要件

### パフォーマンス
- ツール説明文リンティング: 100ツール以下のMCPサーバーに対して5秒以内に完了
- トレースデータの記録: ツール呼び出しへのオーバーヘッドが50ms以下
- レポート生成: 1000トレースレコードから10秒以内に生成

### ユーザビリティ（エージェントにとっての使いやすさ）
- MCPサーバーとして動作するため、既存のMCPクライアント（Claude Code等）からそのまま利用可能
- 全ツールの戻り値が構造化JSONで、エージェントがパースせずに判断に使える
- エラーメッセージは原因と推奨アクションを含み、エージェントが次のアクションを自律的に決定できる
- テストシナリオのYAML/JSON形式は、エージェントが自動生成できるシンプルさ

### 信頼性
- トレースデータの欠損ゼロ（記録中のクラッシュでも部分データを保持）

### セキュリティ
- テスト対象MCPサーバーへの接続情報を安全に管理（環境変数またはシークレットストアから取得）
- トレースデータに含まれる機密情報のマスキング機能（Post-MVP: P1。MVP段階ではOSのファイルパーミッション600によるアクセス制御に委譲）
- MCP Gauge自身が対象サーバーに対して破壊的操作を実行しないセーフガード

### テスト品質
- コードカバレッジ目標: エンジンレイヤー 90%以上、その他レイヤーの詳細はアーキテクチャ設計書に定義
- テスト結果の再現性: 同一条件での実行で決定論的チェック項目の結果が100%一致

### スケーラビリティ
- 単一MCPサーバーに対するテストを想定（MVP段階）
- トレースデータの保存: ローカルファイルシステム（SQLite）で10万レコードまで対応

## スコープ外

明示的にスコープ外とする項目:
- MCPサーバーの実装自体（コード生成やスキャフォールディングはエージェントの責務）
- MCPクライアントのテスト
- LLMモデル自体の評価・ベンチマーク
- リアルタイムモニタリング・アラート（バッチ実行のみ）
- GUIダッシュボード（MVP段階ではMCPツール経由の構造化JSON出力のみ）
- 課金・ライセンス管理機能
