# MCP Gauge — MCPサーバーをE2EテストするためのMCPサーバー

## 概要

コーディングエージェント（Claude等）がMCPサーバーの開発者であると同時にテスターでもあるという前提に立ち、エージェント自身がツールとして利用しながら対象MCPサーバーをE2Eで検証・デバッグするためのMCPサーバー。

従来のAPIテストが決定論的な入出力を前提とするのに対し、LLMエージェントの非決定的な振る舞い（パラメータの組み合わせ、呼び出し順序、リトライ戦略）を考慮したテスト・評価を実現する。

## 解決する課題

- MCPサーバーの品質評価がLLMの実際の利用パターンに基づいていない
- Promptsやツール説明文の最適化に定量的フィードバックがない
- LLMのツール呼び出しをMCPサーバー開発者側からトレーシングする手段がない（既存ツールはクライアント側への計装が前提）
- MCPサーバーの変更に対する回帰テストが手動に依存している

## コア機能

### 1. LLM行動トレーシング

MCPプロトコルの内側から、クライアント実装に依存せずにトレースデータを収集する。

**計測メトリクス：**

- タスク完了までのツール呼び出し回数
- 同一ツールの冗長な再呼び出し数
- ツール間の呼び出し順序の最適性（不要な往復の検出）
- エンドツーエンドの所要時間
- エラーからのリカバリに要した追加ステップ数
- タスク成功率

### 2. Prompts品質評価

MCPのPromptsがツール群の効率的な利用を導けているかを定量評価する。

- Prompts変更前後でのステップ数比較（A/Bテスト）
- LLMが誤ったパラメータでツールを呼んだ回数（説明不足の指標）
- ツールディスカバリ後にLLMが使わなかったツールの検出（発見しにくいPromptsの指標）
- エージェントのセルフリフレクションによる改善提案（トレースデータを元にエージェント自身が分析）

### 3. セキュリティ・安全性検証

エージェントが意図しない形でツールを使うリスクを検出する。

- 破壊的操作（apply/destroy等）が前提条件なしに呼ばれる経路の探索
- ユーザー入力がCLIコマンド等にそのまま渡されるインジェクションパターンの検出
- ツール群の組み合わせで危険な操作に到達できるパスのファジング
- 権限境界の検証（読み取り専用であるべきフローから書き込み操作に到達できないか）

### 4. ツール説明文リンティング

各ツールのdescriptionフィールドの品質を静的に分析する（LLM呼び出し不要）。

- 曖昧な表現の検出（例：「適切な値を指定」→ LLMには判断不能）
- 必須/任意パラメータの説明充足度チェック
- 戻り値の構造がdescriptionに記載されているかの検証
- CIパイプラインに組み込み可能な軽量チェックとして提供

### 5. シナリオレコーダー / リプレイ

実際のエージェント対話セッションをキャプチャし、テスト資産として活用する。

- 実セッションのツール呼び出しシーケンスを記録
- 記録したシナリオをMCPサーバー変更後にリプレイして回帰テスト
- 手動テストシナリオ作成コストの大幅削減
- 「本番で起きたフロー」をそのままテストケース化

### 6. マルチクライアント互換性テスト

異なるLLMモデルでの互換性を検証する。

- 同一シナリオをClaude / GPT / Gemini等で実行し結果を比較
- モデル固有のツール呼び出しの癖による失敗パターンの検出
- 「Claudeでは3ステップで完了するがGPTでは失敗する」等の互換性レポート生成

### 7. コスト推定・最適化レコメンデーション

テスト実行時のトークン消費を可視化し、設計改善につなげる。

- 各シナリオの推定トークン消費量を記録
- Prompts/ツール設計の変更がコストに与える影響の可視化
- 冗長な戻り値の検出と要約形式への変更レコメンデーション
- 「この変更で1タスクあたり推定○○トークン削減」の定量提示

### 8. コントラクトテスト

MCPサーバー間の依存関係における契約検証。

- サーバー間のインターフェース契約（入出力スキーマ）の検証
- 上流サーバーの変更が下流サーバーに与える影響の検出
- マイクロサービスにおけるPactに相当する位置づけ

## 新規性

| 観点 | 評価 | 根拠 |
|------|------|------|
| LLM視点のテスト | ★★★ | LLMが実際にどうツールを使うかに基づくテストは既存ツールにない |
| MCPプロトコル内部からの観測 | ★★★ | クライアント側計装不要のトレーシングは差別化ポイント |
| Prompts最適化の定量評価 | ★★★ | この領域に特化したツールは現時点で存在しない |
| セキュリティファジング | ★★☆ | 概念自体は既知だがMCPコンテキストでの適用は新しい |
| プロトコルレベルテスト | ★☆☆ | MCP Inspector等で一部カバー可能 |

## プロダクトポジショニング

単なる「MCPサーバーのデバッグツール」ではなく、**MCPサーバーのDeveloper Experienceプラットフォーム**として位置づける。MCPエコシステムの成長に伴い「良いMCPサーバーの作り方」の需要が高まるタイミングと合致する。

## 想定される課題

- **評価の信頼性**: テスターもLLMであるため、テスト自体の正しさの保証が難しい → LLM判断と決定論的アサーションのハイブリッド設計で対応
- **コスト**: E2Eテストの頻繁な実行にはLLM APIコストがかかる → 軽量な静的チェック（リンティング）とLLMベーステストの階層化
- **非決定性**: LLMの応答のばらつきにより同一テストでも結果が変わりうる → 複数回実行の統計的評価、決定論的チェックポイントの併用

## 開発アプローチ

まずGalley自身のMCPサーバーをテスト対象としたプロトタイプを構築し、自らがヘビーユーザー（dogfooding）となることで設計精度を高める。プリセールスデモという日常業務で発生する実フローをシナリオレコーダーで蓄積し、テスト資産を自然に成長させる。
